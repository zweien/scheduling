# 【功能请求】账号管理页支持管理员修改成员密码

## 功能需求

在账号管理页面中，管理员除了查看账号、调整角色、启用/停用账号外，还应支持直接为指定成员重置或修改密码。

## 设计目标

- 管理员在账号管理页内完成成员密码修改，不增加额外页面跳转
- 保留成员在设置页中修改自己密码的现有能力，两条路径并存
- 仅管理员可以执行“修改任意成员密码”操作
- 修改成功后给出明确成功反馈，并写入审计日志

## 方案选择

### 方案 1：账号管理页卡片按钮 + 管理员专用弹窗

推荐方案。

- 在账号管理页每个账号卡片上新增“修改密码”按钮
- 点击后打开管理员专用弹窗，输入新密码和确认密码
- 通过管理员专用 action 更新指定账号密码

优点：

- 交互路径最短，符合管理员账号维护场景
- 与现有账号管理页结构一致，改动集中
- 复用底层密码更新能力，不需要新建复杂流程

缺点：

- 需要补一个管理员专用 action 和弹窗组件

### 方案 2：复用设置页密码修改区域，加账号选择器

不推荐。

- 会把“修改自己密码”和“管理员维护他人账号”混在同一页面
- 使用场景不自然，增加页面认知负担

### 方案 3：在账号卡片内直接展开密码输入框

不推荐。

- 密码输入长期暴露在列表中，界面噪音大
- 容易误操作，也不利于保持组件职责清晰

## 交互设计

### 入口

- 账号管理页每个账号卡片增加 `修改密码` 按钮
- 管理员对所有账号都可操作，包括自己的账号

### 弹窗内容

- 标题：`修改成员密码`
- 目标账号信息：
  - 显示名称
  - 用户名
- 输入项：
  - 新密码
  - 确认新密码

### 校验规则

- 新密码不能为空
- 新密码长度至少 6 位
- 两次输入必须一致

### 成功行为

- 提交成功后关闭弹窗
- 页面弹出成功提示
- 审计日志记录操作者和被修改账号

### 失败行为

- 表单内显示错误信息
- 不关闭弹窗
- 不刷新账号列表

## 代码落点

### Action 层

在 [src/app/actions/accounts.ts](/home/z/scheduling/src/app/actions/accounts.ts) 新增管理员专用 action，例如：

- `adminUpdateAccountPasswordAction(accountId, newPassword)`

职责：

- `requireAdmin()`
- 校验密码长度
- 检查目标账号是否存在
- 调用 [src/lib/accounts.ts](/home/z/scheduling/src/lib/accounts.ts) 中现有的 `updateAccountPassword(accountId, newPassword)`
- 写审计日志

### 组件层

在 [src/components/AccountManagement.tsx](/home/z/scheduling/src/components/AccountManagement.tsx) 中：

- 新增“修改密码”按钮
- 维护当前选中的目标账号和弹窗开关

新增管理员密码弹窗组件，例如：

- `src/components/AdminAccountPasswordDialog.tsx`

职责：

- 展示目标账号信息
- 收集新密码和确认新密码
- 调用管理员 action
- 处理成功提示和错误提示

## 权限边界

- 仅管理员可以访问账号管理页，因此管理员专用密码修改入口默认也只会对管理员可见
- 普通用户仍通过设置页的个人密码修改入口修改自己的密码
- 不要求管理员输入旧密码

## 审计日志

- 继续复用已有 `set_password` 日志动作
- 不记录密码明文
- 记录内容至少包括：
  - 操作者用户名
  - 被修改账号用户名

## 测试策略

### Node / Action 测试

- 管理员可以修改指定账号密码
- 目标账号不存在时返回错误
- 密码少于 6 位时返回错误
- 非管理员调用被拒绝

### Playwright

- 账号管理页显示“修改密码”按钮
- 点击后可打开弹窗
- 输入合法密码后修改成功，并显示成功提示
- 被修改成员可使用新密码登录

### 回归验证

- 设置页中“修改自己密码”的旧流程保持可用
- 账号角色调整、启停账号、创建账号不受影响
