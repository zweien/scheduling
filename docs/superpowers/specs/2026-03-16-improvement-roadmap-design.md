# 项目改进路线图设计

日期：2026-03-16

## 1. 目标

在不打断当前业务可用性的前提下，分阶段收敛项目的权限边界、工程基线、组件职责、数据库迁移和发布流程。

本路线图的目标不是一次性重构，而是用多次小步迭代把项目从“功能闭环”推进到“可持续维护”。

## 2. 分阶段原则

- 每一阶段都必须独立可测试、可回退
- 优先处理“行为不一致”和“会阻塞后续开发”的问题
- 先收敛基线，再做结构性优化
- 不在同一阶段混合无关主题

## 3. 阶段划分

### 阶段 1：权限与工程基线

目标：

- 修正值班人员页普通用户只读的数据权限错层
- 清理当前全量 lint 阻塞项
- 补一轮权限与关键页面回归

原因：

- 当前最接近真实 bug 的问题是“页面权限”和“数据权限”不一致
- 当前 `npm run lint` 不是全绿，继续扩展会持续积累工程债

产出：

- 读写权限边界更清晰
- 全量 lint 恢复可通过
- 浏览器回归更稳定

### 阶段 2：值班人员管理模块拆分

目标：

- 拆分值班人员管理页的大组件
- 拆分 `users` action 的职责边界

原因：

- 当前值班人员管理已经同时承担列表、筛选、导入、编辑、权限裁剪
- 再继续加功能会快速恶化可维护性

产出：

- 更清晰的组件边界
- 更稳定的测试和迭代速度

### 阶段 3：数据库迁移机制版本化

目标：

- 引入最小可用 migration 版本管理
- 逐步替换 `db.ts` 中的隐式 `ALTER TABLE try/catch`

原因：

- 当前数据库初始化方式适合早期，但不适合持续演进
- release 驱动部署已经建立，数据库版本管理需要跟上

产出：

- 可追踪的 schema 变更历史
- 更安全的部署和回滚路径

### 阶段 4：发布流程完善

目标：

- 完善 release/tag 规范
- 补齐发布文档和回滚说明

原因：

- 当前 workflow 已切到 release 驱动，但流程规范还不完整

产出：

- 更清晰的发版路径
- 更可控的线上变更

## 4. 当前建议顺序

推荐按以下顺序推进：

1. 阶段 1：权限与工程基线
2. 阶段 2：值班人员管理模块拆分
3. 阶段 3：数据库迁移机制版本化
4. 阶段 4：发布流程完善

## 5. 风险控制

- 阶段 1 和阶段 2 应尽量避免数据库结构变更
- 阶段 3 独立处理数据库迁移，不混入 UI 改动
- 阶段 4 只动流程与文档，不混业务逻辑

## 6. 第一阶段实现范围

第一阶段仅包含：

- 修复普通用户值班人员只读查询的后端权限边界
- 修复当前全量 lint 阻塞项
- 运行并稳定关键权限和页面回归

明确不包含：

- 不拆组件
- 不改数据库迁移机制
- 不改 release 流程
