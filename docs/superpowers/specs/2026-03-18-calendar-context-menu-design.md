# 排班月历右键菜单设计

## 背景

当前月历日期格只有左键点击入口：

1. 空日期左键打开人员选择弹窗，安排值班人员
2. 已有排班日期左键打开弹窗，可替换、移动或删除
3. 桌面端还支持拖拽移动/交换

这套能力已经可用，但桌面端缺少一个更符合操作习惯的右键入口。尤其在月历视图里，用户会自然预期可以通过右键直接看到当前日期的可执行操作。

本次需求是在排班月历视图中为桌面端增加日期右键菜单，并在空日期菜单里加入一个“自动排班”工具入口。

## 本次范围

1. 仅在桌面端月历视图支持右键菜单
2. 空日期右键菜单提供：
   - `自动排班`
   - `安排值班人员`
3. 已有排班日期右键菜单提供：
   - `替换值班人员`
   - `移动到其他日期`
   - `删除排班`
4. `自动排班` 点击后弹出对话框，可设置连续天数和排班起点模式

明确不包含：

1. 不处理移动端长按菜单
2. 不替换现有左键点击和拖拽交互
3. 不重做 `UserSelectDialog`
4. 不支持自动排班时跳过已有日期或覆盖已有日期

## 目标

1. 为桌面端提供更高效的日期操作入口
2. 保持现有左键和拖拽能力不变
3. 自动排班能力能够以“从当前日期开始连续安排 N 天”的方式局部生效

## 非目标

1. 不建立一套完整的全局 context menu 基础设施
2. 不把所有日历操作都迁移到右键菜单
3. 不扩展到列表视图、打印页或导出页

## 方案对比

### 方案 A：右键菜单只做入口，动作复用现有流程

右键菜单负责显示菜单项和当前日期上下文，真正动作继续回到 `CalendarView` 的现有逻辑。

优点：

- 改动最小
- 和现有弹窗、移动模式、删除逻辑保持一致
- 符合 DRY

缺点：

- 需要为“自动排班”新增一条最小 server action

### 方案 B：右键菜单直接承载所有操作流程

例如直接在右键菜单里嵌套人员选择、移动确认等。

优点：

- 操作路径更短

缺点：

- 菜单会迅速膨胀
- 复制现有弹窗和状态逻辑，维护成本高

### 方案 C：重做完整 context menu 体系

优点：

- 扩展性看起来最好

缺点：

- 明显过度设计，不符合当前需求规模

## 选型

采用方案 A。

原因：

- 右键菜单本质只是新增一个桌面端入口
- 已有左键和拖拽流程已经能工作，不需要重构
- 唯一新增业务能力是“连续自动排班”，可以独立成一条最小 server action

## 交互设计

### 右键触发规则

1. 仅桌面端生效
2. 有管理权限时才显示
3. 右键时阻止浏览器原生菜单
4. 菜单在鼠标坐标附近弹出
5. 点击空白处、按 `Esc`、执行菜单项后关闭

### 空日期菜单

- `自动排班`
- `安排值班人员`

### 已有排班日期菜单

- `替换值班人员`
- `移动到其他日期`
- `删除排班`

## 自动排班对话框设计

### 字段

1. 起始日期
   - 只读展示
2. 连续天数
   - 数字输入
   - 必填
   - 最小值 `1`
   - 默认值为“当前启用中的值班人员总数”
3. 排班起点
   - 单选项
   - 选项一：`延续现有排班（推荐）`
   - 选项二：`从首位人员开始`

### 排班起点语义

#### 延续现有排班

从起始日前最近一个已有排班人员的下一个人开始排。

如果起始日前没有任何排班记录，则回退到“从首位人员开始”。

#### 从首位人员开始

从当前启用人员顺序的第一位开始排。

### 执行规则

1. 只允许对空日期触发
2. 只使用启用中的值班人员
3. 从起始日期开始连续安排 `N` 天
4. 范围内如果任意日期已有排班，直接阻止执行并报错
5. 首版不做跳过、不做覆盖

## 组件设计

### CalendarCell

新增 `onContextMenu` 透传。

职责仍保持单一：

- 呈现日期格
- 转发左键、拖拽、右键事件

### CalendarView

这是本次主要改动点。

新增一个最小右键菜单状态：

- `date`
- `hasSchedule`
- `x`
- `y`

可选再带当前 `schedule`，避免二次查找。

`CalendarView` 负责：

1. 打开/关闭右键菜单
2. 根据当前日期触发已有操作流程
3. 管理新增的自动排班弹窗状态

### CalendarContextMenu

新增一个轻量组件，只负责：

1. 按坐标定位
2. 根据 `hasSchedule` 渲染不同菜单项
3. 点击菜单项后向上抛出动作

不直接触达业务 action。

### AutoScheduleDialog

新增一个小对话框组件，只负责：

1. 展示起始日期
2. 收集连续天数
3. 收集排班起点模式
4. 提交 / 取消

不承担具体排班算法。

## 后端设计

### 新 action

新增一条 server action，例如：

- `autoScheduleFromDate(startDate, days, startMode)`

职责：

1. 校验权限
2. 校验起始日期为空日期
3. 校验连续天数
4. 校验范围内不存在已有排班
5. 基于启用人员顺序生成连续排班
6. 写日志
7. `revalidatePath('/dashboard')`

### 新算法 helper

建议在 `src/lib/schedule.ts` 新增一个专用 helper，而不是把现有 `generateSchedule()` 改成胖接口。

建议接口形态：

- `generateScheduleFromDate(startDate, days, startMode)`

其中：

- `startMode = 'continue' | 'from_first'`

计算规则：

1. 取启用人员顺序列表
2. 若 `from_first`
   - 从 `users[0]` 开始
3. 若 `continue`
   - 查询起始日前最近一条排班
   - 找到对应人员在当前启用顺序中的位置
   - 从下一个人开始
   - 若找不到或没有历史排班，则回退到首位

## 日志设计

新增一条日志动作：

- `auto_schedule_from_date`

日志内容建议包含：

1. 起始日期
2. 连续天数
3. 起点模式
4. 实际生成日期范围

## 事件边界

1. 右键不触发现有左键点击弹窗
2. 正在拖拽时不打开右键菜单
3. 正在移动模式时右键菜单默认关闭，避免双重状态
4. 打开右键菜单时关闭上一个右键菜单
5. 右键菜单动作执行后关闭菜单

## 测试策略

### Playwright 桌面端回归

至少覆盖：

1. 右键空日期显示：
   - `自动排班`
   - `安排值班人员`
2. 右键已有排班日期显示：
   - `替换值班人员`
   - `移动到其他日期`
   - `删除排班`
3. 右键空日期选择 `自动排班` 后弹出对话框
4. 默认连续天数等于启用人员数量
5. 切换排班起点模式后可成功生成连续排班

### 服务层测试

至少覆盖：

1. `continue` 模式下从最近排班的下一人开始
2. `from_first` 模式下从首位人员开始
3. 范围内有已有排班时拒绝执行
4. 没有启用人员时拒绝执行

## 风险与控制

### 风险 1：右键菜单和左键/拖拽交互冲突

控制：

- 菜单只作为入口
- 明确在打开菜单时处理拖拽和移动模式边界

### 风险 2：自动排班范围误伤已有排班

控制：

- 首版只允许“整段范围都为空”时执行
- 不做覆盖和跳过

### 风险 3：顺序延续逻辑不清晰

控制：

- 把起点模式显式交给用户选择
- 默认推荐“延续现有排班”

## 实施顺序建议

1. 先补右键菜单和自动排班的失败测试
2. 再补 `CalendarContextMenu` 和 `AutoScheduleDialog`
3. 最后补后端自动排班 action 与联合回归
