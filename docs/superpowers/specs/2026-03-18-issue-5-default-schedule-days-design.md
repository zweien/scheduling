# Issue #5 设计：管理员可配置默认排班天数

## 背景

当前排班生成入口在只填写开始日期、不填写结束日期时，会走后端自动推导逻辑。现有界面文案按“启用中的值班人员数”提示默认排班天数，但管理员无法在后台显式配置这一默认值，和实际业务需求不匹配。

Issue #5 要求新增一个管理员可配置的“默认排班天数”设置，并让未填写结束日期时统一按该值生成排班；如果用户手动填写结束日期，则继续以手动输入为准。

## 目标

1. 管理员可在设置页修改默认排班天数
2. 未填写结束日期时，后端按配置值推导排班结束日期
3. 手动填写结束日期时，继续覆盖默认规则
4. 修改配置写入操作日志
5. 保持现有注册设置和密码修改等设置页结构不被破坏

## 非目标

1. 不按组织、类别或角色分别配置默认排班天数
2. 不新增一套独立的系统设置表或复杂设置中心
3. 不改变已有手动指定结束日期的生成逻辑
4. 不扩展到自动排班右键弹窗等其他天数默认值来源

## 现状分析

当前仓库已经有最小配置存储能力：

- `src/lib/config.ts` 通过 `config` 表读写键值配置
- `registration_enabled` 已用这套模式落地
- `src/app/actions/config.ts` 已有管理员配置 action 和日志写入模式
- `src/app/dashboard/settings/page.tsx` 已有管理员专属设置区块

这意味着本需求不需要新建 `settings` 表。直接在现有 `config` 表增加 `default_schedule_days` 键即可，范围更小，也更符合 KISS。

## 方案对比

### 方案 A：复用现有 `config` 表，新增 `default_schedule_days` 键

优点：

- 复用现有配置存储和设置页模式
- 改动集中，风险最低
- 和 `registration_enabled` 的实现方式一致

缺点：

- `config` 仍是字符串键值，需要专用封装保证类型安全

### 方案 B：新建独立系统设置表

优点：

- 看起来更“规范”

缺点：

- 明显过度设计
- 和当前仓库已有配置模式重复

### 方案 C：只在前端本地配置默认值

优点：

- 实现最快

缺点：

- 不满足“管理员后台可配置”
- 前后端规则容易分叉

## 结论

采用方案 A：复用现有 `config` 表，新增 `default_schedule_days` 作为全局系统设置。

## 详细设计

### 1. 配置存储

在 `src/lib/config.ts` 中新增专用封装：

- `getDefaultScheduleDays()`
- `setDefaultScheduleDays(days: number)`

行为约束：

- 首次未配置时返回一个内置默认值
- 首版建议默认值为 `21`

原因：

- Issue 原文明确说“当前固定为 21 天，无法调整”
- 用 `21` 作为未配置回退值，能保持旧版本行为不变

### 2. 管理员设置入口

在设置页新增管理员可见卡片，例如“排班设置”或“默认排班天数”。

页面行为：

- 展示当前默认排班天数
- 提供数字输入框和保存按钮
- 仅管理员可见
- 保存成功后显示成功提醒
- 保存失败时显示错误提示

实现模式复用现有 `RegistrationSettings`：

- 服务端读取初值
- 客户端局部状态编辑
- 提交 action 保存

### 3. 后端生成逻辑

当前 `generateScheduleAction(startDate, endDate?)` 在 `endDate` 为空时交给底层生成逻辑自动推导。

本次要求把“默认排班天数”规则统一收敛到后端：

- 如果传入 `endDate`
  - 继续按 `startDate ~ endDate` 生成
- 如果未传入 `endDate`
  - 读取 `default_schedule_days`
  - 由后端推导真实结束日期
  - 再执行生成

这样可以避免前端只显示提示但不真正参与规则决策，保证任何入口都遵守同一默认值。

### 4. 前端提示更新

`src/components/ScheduleGenerator.tsx` 当前在未填写结束日期时显示：

- “不填将排 X 天”

这里的 `X` 需要改为当前配置值，而不是启用人员数。

界面行为：

- 组件初始化时读取默认排班天数
- 未填写结束日期时展示“不填将默认排 X 天”
- 填写结束日期后隐藏该提示

### 5. 日志

修改默认排班天数需要写日志。

建议：

- 新增日志动作：`update_default_schedule_days`
- `target`：`默认排班天数`
- `old_value`：旧值
- `new_value`：新值

这样日志页能准确反映管理员配置变更。

## 错误处理

### 配置更新

- 输入为空、非数字或小于 1 时，前端直接阻止提交
- 后端仍做二次校验，避免绕过前端

### 生成排班

- 若默认值配置异常，后端回退到安全默认值 `21`
- 不让错误配置导致生成入口整体不可用

## 测试策略

### 单元/服务层

1. 未配置 `default_schedule_days` 时返回 `21`
2. 配置后读取为最新值
3. 非法输入会被拒绝
4. 未传 `endDate` 时按配置值推导生成范围
5. 显式传入 `endDate` 时不受配置值影响

### Playwright

1. 管理员在设置页修改默认排班天数成功
2. 返回排班生成入口后，提示文案显示新的默认值
3. 只填开始日期生成时，实际生成天数等于配置值

## 风险与控制

### 风险 1：前后端默认值不一致

控制：

- 默认值只在 `config` 专用封装中定义一次
- 前端提示通过 action 读取，不自己写死

### 风险 2：改动影响现有生成逻辑

控制：

- 只在“未传结束日期”分支接入配置
- 显式传入结束日期的逻辑保持不变

## 预期产出

1. 管理员可在设置页配置默认排班天数
2. 默认排班天数配置写入日志
3. 排班生成入口未填写结束日期时按配置值执行
4. 回归测试覆盖配置、生成和展示链路
