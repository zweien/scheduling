# Issue #4 设计：拖动换班强制填写理由并展示原始/最终值班员

## 背景

当前月历视图支持管理员通过拖拽在日期之间移动或交换排班，但拖拽动作会立即落库，不要求填写调整理由，也无法在日历中直接区分最初值班员和最终值班员。

这带来两个问题：

1. 人工调班缺少强制理由，日志追溯信息不足。
2. 多次拖拽后，页面只能看到当前值班员，无法直观看到该日期最初由谁值班。

## 本次范围

本次仅覆盖月历中的拖拽调整流程：

- 已排班日期拖到空日期：移动排班
- 已排班日期拖到已有排班日期：交换排班

明确不包含：

- 点击日期后手动选择人员替换
- 列表视图的调整交互
- 导出、打印页面展示原始/最终值班员
- 历史调整链路的独立历史表

## 目标

1. 拖拽移动或交换排班时，必须填写理由后才能保存。
2. 日历中能区分最初值班员和最终值班员。
3. 所有拖拽调整都完整记入日志，并单独保存理由。
4. 多次调班后，日历只展示“最初值班员”和“当前最终值班员”。

## 非目标

1. 不改造非拖拽的手动换人流程。
2. 不增加完整调班历史子表。
3. 不在首版中扩展列表页、打印页、导出文件的展示。
4. 不引入额外审批、评论或附件机制。

## 方案对比

### 方案 A：拖拽后进入理由确认，再落库

拖拽命中目标日期后，先弹出理由对话框，确认后再执行移动或交换。

优点：

- 最符合“理由必填”的业务语义。
- 取消时无需回滚数据库。
- 状态边界清晰，便于测试。

缺点：

- 需要维护一个“待确认的拖拽动作”状态。

### 方案 B：先执行拖拽，再补理由，未填写则回滚

优点：

- 拖拽反馈最直接。

缺点：

- 实现复杂，需要处理回滚和并发覆盖。
- 日志和数据库一致性风险高。

### 方案 C：拖拽前先填写理由

优点：

- 后端逻辑简单。

缺点：

- 交互顺序违背用户习惯，体验差。

## 选型

采用方案 A。

原因：

- 满足理由必填的约束且实现成本最低。
- 保证数据库只在用户确认后变更，避免回滚复杂度。
- 便于保持 `drop` 负责识别动作、对话框负责确认、server action 负责落库的单一职责划分。

## 交互设计

### 拖拽流程

1. 管理员从一个已排班日期开始拖拽。
2. 拖到目标日期后，不立即写库，只识别动作：
   - 目标为空：移动
   - 目标已有排班：交换
3. 打开理由对话框，显示：
   - 操作类型：移动排班 / 交换排班
   - 来源日期
   - 目标日期
   - 涉及人员摘要
4. 用户填写理由后点击确认，才真正执行写库。
5. 用户取消或关闭对话框，不做任何数据修改。

### 理由约束

- 必填
- 首版长度限制：10-200 字
- 空白字符需要先 `trim()`

### 异常和取消

- 拖到非日历区域：直接取消，不弹窗
- 拖回原日期：直接取消，不弹窗
- 理由校验失败：对话框保持打开并提示
- 写库失败：关闭 loading，保留对话框或给出错误提示，不显示成功状态

## 数据模型设计

### schedules 表新增字段

新增：

- `original_user_id INTEGER`
- `adjust_reason TEXT`

含义：

- `original_user_id`：该日期最初的值班人员。首次生成或首次设置排班时，值等于当前 `user_id`。
- `adjust_reason`：最近一次拖拽调整成功时填写的理由。

规则：

- 创建新排班时，如果没有 `original_user_id`，默认写成当前 `user_id`
- 后续拖拽移动或交换时：
  - `original_user_id` 只在为空时补齐，不被覆盖
  - `adjust_reason` 更新为本次拖拽理由

### logs 表新增字段

新增：

- `reason TEXT`

用途：

- 单独保存这次换班或移动的理由，不和 `old_value` / `new_value` 混写

### 为什么不建历史表

本次需求只要求：

- 日历展示最初和最终值班员
- 日志记录每一步操作和理由

日志已经足够承载完整追踪；再增加历史表会显著扩大改动面，超出当前范围。

## 展示设计

### 月历单元格

未发生拖拽调整：

- 保持当前单人显示模式不变

发生拖拽调整且 `original_user_id !== user_id`：

- 桌面端显示为两行摘要
  - `原：XXX`
  - `现：XXX`
- 移动端保持简化展示，避免格子拥挤
- tooltip 可补充完整人员信息

### 手动调整标记

继续保留“手动调整”标记。

其语义收敛为：

- 发生过人工替换或拖拽调整的日期

## 服务端设计

### action 改造

当前拖拽相关 action 需要增加理由参数：

- `moveSchedule(fromDate, toDate, reason)`
- `swapSchedules(fromDate, toDate, reason)`

行为要求：

1. 未传理由或理由不合法时拒绝执行。
2. 成功后更新目标日期排班信息。
3. 对涉及的排班记录维护：
   - `user_id`
   - `original_user_id`
   - `adjust_reason`
   - `is_manual`
4. 写入对应日志，并把 `reason` 落到日志表。

### 日志动作

建议使用更明确的动作名：

- `move_schedule`
- `swap_schedule`

日志内容：

- `target`：包含来源日期、目标日期
- `old_value`：调整前人员摘要
- `new_value`：调整后人员摘要
- `reason`：本次填写的理由

## 前端组件设计

### CalendarView

新增一个待确认拖拽状态，至少包含：

- `sourceDate`
- `targetDate`
- `type: 'move' | 'swap'`
- `sourceUserName`
- `targetUserName | null`

职责调整：

- `onDrop`：只负责识别动作并打开对话框
- 对话框确认：才调用对应 server action
- `dragend`：继续负责清理拖拽视觉状态

### 新增理由对话框组件

建议新增独立组件，例如：

- `ScheduleAdjustmentReasonDialog`

职责：

- 展示操作摘要
- 收集理由输入
- 处理确认/取消
- 不直接持有业务数据源

这样可以避免把 `CalendarView` 继续堆成大组件。

## 迁移策略

新增数据库迁移，步骤：

1. 给 `schedules` 增加 `original_user_id`
2. 给 `schedules` 增加 `adjust_reason`
3. 给 `logs` 增加 `reason`
4. 对现有 `schedules` 做一次回填：
   - `original_user_id = user_id`，仅对空值记录执行

这样历史数据无需人工处理，现有日历也能平滑兼容。

## 测试策略

### 数据层 / action 测试

覆盖：

1. 移动排班需要理由
2. 交换排班需要理由
3. 首次拖拽后：
   - `original_user_id` 保持初始人员
   - `adjust_reason` 写入
4. 多次拖拽后：
   - `original_user_id` 不变
   - `user_id` 为最终人员
5. 日志写入：
   - 动作名正确
   - `reason` 正确保存

### Playwright 交互测试

覆盖：

1. 拖到目标日期后弹出理由对话框
2. 不填理由无法确认
3. 取消后不落库
4. 填写理由确认后，日历更新
5. 调整后的日期显示“原/现”摘要

### 日志页面回归

至少验证：

1. 有一条移动日志包含理由
2. 有一条交换日志包含理由

## 风险与控制

### 风险 1：月历格信息过载

控制：

- 只在发生调整时显示“原/现”
- 移动端保持简化展示

### 风险 2：拖拽交互复杂度增加

控制：

- 只在拖拽确认链路中新增一个对话框
- 不改造非拖拽路径

### 风险 3：旧数据没有 `original_user_id`

控制：

- 迁移时回填当前 `user_id`

## 实施顺序建议

1. 数据库迁移与类型定义
2. server action 改造与日志写入
3. 理由对话框与 `CalendarView` 两阶段确认
4. 月历“原/现”展示
5. 自动化测试与回归
