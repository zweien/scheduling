# Issue #24 第一期设计：月历多选底部浮动操作栏

## 背景

当前月历视图已经支持：

- 多选日期
- 全选当月
- 取消选择
- 批量删除

但这些能力分散在顶部操作区，且只覆盖删除。随着月历交互逐步增强，管理员在多选后缺少一个集中、稳定的批量操作入口，操作效率不高，也不便继续扩展更多批量动作。

## 目标

本次只完成 `#24` 的第一期能力：

1. 在月历视图中增加底部浮动操作栏
2. 对已选日期提供 3 个批量动作：
   - 批量编辑为同一值班人员
   - 导出已选日期
   - 批量删除

## 非目标

本次明确不包含：

- 空状态引导
- 搜索高亮
- 列表视图的多选批量操作
- 多格式导出选择
- 复杂的批量编辑规则，例如按日期分别指定不同人员

## 交互设计

### 1. 触发条件

仅在以下条件同时满足时显示底部浮动操作栏：

- 当前为月历视图
- `canManage = true`
- `selectedDates.size > 0`

### 2. 浮动操作栏内容

浮动栏显示：

- 已选日期数量
- `批量编辑`
- `导出已选`
- `批量删除`

其中：

- `批量编辑` 使用主操作样式
- `导出已选` 使用次级样式
- `批量删除` 使用危险操作样式

### 3. 与顶部操作区的关系

顶部现有多选辅助操作保留：

- `全选当月`
- `取消选择`

真正的批量动作收敛到底部浮动栏，不再在顶部重复出现批量删除，避免同一操作出现两处入口。

## 功能行为

### 批量编辑

- 点击后打开简化的人员选择弹窗
- 用户选定一个值班人员后，将所有已选日期统一替换为该人员
- 成功后：
  - 刷新月历
  - 清空选中状态
  - 显示成功提示

首版规则：

- 所有已选日期统一替换为同一人员
- 不支持针对不同日期分别指定不同人员

### 导出已选

- 仅导出已选日期
- 首版只提供 `XLSX`
- 已选日期可以是不连续的

导出语义必须与“按连续日期范围导出”分开，不能把未选日期一并导出。

### 批量删除

- 继续复用当前批量删除确认逻辑
- 成功后清空选中状态并刷新月历

## 实现设计

### 1. 组件拆分

新增一个轻量组件，例如：

- [src/components/SelectedSchedulesActionBar.tsx](/home/z/scheduling/src/components/SelectedSchedulesActionBar.tsx)

职责：

- 展示已选数量
- 渲染 3 个动作按钮
- 将动作回调交还给上层

不直接承担任何业务逻辑，保持单一职责。

### 2. 页面集成

在 [src/components/CalendarView.tsx](/home/z/scheduling/src/components/CalendarView.tsx) 集成：

- 继续持有 `selectedDates`
- 控制浮动栏显示与隐藏
- 处理批量编辑、导出、删除动作

### 3. 批量编辑 Action

新增一个专用 action，例如：

- `replaceSchedules(dates: string[], userId: number)`

设计原因：

- 不应在前端循环调用单日 `replaceSchedule`
- 批量操作应该有独立的日志语义
- 错误处理和结果统计更一致

日志建议：

- 动作名：`batch_replace_schedules`
- 内容：记录日期数量、具体日期列表和目标人员

### 4. 已选导出 Action

新增一个专用 action，例如：

- `exportSelectedSchedulesToXlsx(dates: string[])`

设计原因：

- 现有导出基于连续日期范围
- 已选日期天然可能不连续
- 不应为了复用现有接口而错误导出未选日期

## 测试策略

### Node / Action 层

- 批量编辑只更新已选日期
- 批量编辑写入正确日志
- 已选导出只包含已选日期

### Playwright

- 选中日期后底部浮动栏出现
- 批量编辑成功后，多个已选日期统一更新为同一人员
- 批量删除成功后，浮动栏消失且选中状态清空
- 导出已选触发下载

### 回归范围

- 现有多选
- 单日替换
- 当前批量删除
- 月历拖拽与右键菜单

## 方案取舍

本次选择只做 `#24` 第一期，即“底部浮动操作栏”，原因如下：

- 业务价值最高，直接提升管理员多选操作效率
- 技术主线清晰，围绕 `selectedDates` 现有能力扩展即可
- 不把空状态和搜索高亮混在同一轮，避免范围失控

这符合：

- KISS：只解决当前最明确的效率问题
- YAGNI：不提前做整包 UI 改造
- DRY：复用现有月历多选机制，不新造第二套选择状态
