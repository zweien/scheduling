# Issue #3 排班导入设计

日期：2026-03-16

## 1. 目标

为排班模块增加“导入已有值班表”能力，支持模板下载、Excel 上传、预检、冲突处理策略选择和导入结果统计。

本次设计只覆盖首版可用能力，优先满足以下场景：

- 其他部门已经排好班，管理员通过模板导入同步到系统
- 导入前能明确识别格式错误、空值、文件内重复和系统内已有排班冲突
- 针对系统内已有排班，支持“跳过”“覆盖”“标记冲突”三种策略

## 2. 范围

### 本次包含

- 在排班页右上角提供“导入排班”入口
- 提供排班导入模板下载
- 支持上传 `.xlsx` 文件并执行预检
- 预检结果区分格式错误、文件内重复、系统冲突
- 支持选择冲突处理策略：`skip`、`overwrite`、`mark_conflicts`
- 导入完成后展示结果统计
- 记录导入相关审计日志

### 本次不包含

- 逐条人工确认冲突后再继续导入
- 自动创建不存在的值班人员
- 为排班记录新增备注字段或修改数据库表结构
- 支持 `.csv`、`.xls` 等额外格式

## 3. 设计原则

- 复用现有“先校验、后导入”的两阶段模式，避免新造复杂导入框架
- 前端和后端职责分离：前端负责选择策略和展示结果，后端负责预检与落库
- 保持最小必要模板字段，避免为首版引入无关 schema 变更
- 冲突处理以“按日期唯一”作为规则，不引入多余抽象

## 4. 用户流程

1. 管理员在排班页点击“导入排班”
2. 下载模板并填写排班数据
3. 上传文件并点击“校验文件”
4. 系统返回预检结果
5. 如果存在系统内冲突，管理员选择处理策略
6. 点击“开始导入”
7. 系统返回导入结果统计

当选择 `mark_conflicts` 时：

- 仍执行预检
- 展示冲突列表和统计
- 不执行落库

## 5. 模板设计

模板字段保持最小必要集：

- `日期（必填，YYYY-MM-DD）`
- `值班人员姓名（必填）`
- `是否手动调整（必填，是/否）`
- `备注（选填）`

字段说明：

- `日期` 用于唯一定位排班日期
- `值班人员姓名` 用于匹配现有值班人员
- `是否手动调整` 对应 `schedules.is_manual`
- `备注` 首版只做解析和展示，不写入数据库

选择按姓名匹配而不是按 `userId` 匹配，原因是外部部门提供的排班表通常只包含可读姓名，不会知道系统内部主键。

## 6. 冲突定义与处理策略

### 冲突定义

同一日期在系统中已存在排班记录，且导入文件中也提供了该日期的新排班数据，则判定为系统冲突。

### 处理策略

- `skip`
  - 跳过系统中已存在的日期
  - 仅导入无冲突记录
- `overwrite`
  - 使用导入文件中的记录覆盖系统中已有排班
  - 通过现有 `setSchedule()` 完成写入
- `mark_conflicts`
  - 仅返回冲突列表和统计结果
  - 不执行任何写库操作

## 7. 预检规则

预检服务分四层执行：

1. 文件级校验
   - 能否解析为合法 `.xlsx`
   - 是否存在工作表
   - 表头是否与模板一致

2. 行级字段校验
   - 日期不能为空且格式合法
   - 值班人员姓名不能为空
   - 是否手动调整必须是“是”或“否”

3. 文件内一致性校验
   - 同一日期在导入文件中重复出现，视为文件内重复

4. 系统数据校验
   - 值班人员姓名必须能匹配到现有值班人员
   - 日期在系统中已有排班时，记录为系统冲突

规则约束：

- 只要存在格式错误、非法值、人员无法匹配或文件内重复，本次导入必须阻止执行
- 系统冲突本身不阻止预检通过，但必须由策略决定后续行为

## 8. 预检返回结构

新增排班导入预检结果类型，建议至少包含：

- `totalRows`
- `validRows`
- `invalidRows`
- `duplicateRows`
- `conflictRows`
- `cleanRows`
- `rows`
- `issues`
- `conflicts`

其中：

- `issues` 用于承载会阻止导入的错误
- `conflicts` 用于展示“系统已有排班”的可处理冲突
- `rows` 保存通过基础校验后的标准化记录，供后续导入复用

## 9. 前端结构

入口放在排班页右上角，保持与导出、打印等动作同层级。

建议拆分为以下组件：

- `ScheduleImportDialog`
  - 负责弹窗开关、文件选择、策略选择、结果展示
- `ScheduleImportSummary`
  - 展示预检统计和导入结果统计
- `ScheduleImportConflictList`
  - 展示冲突日期、系统原排班、导入目标排班

前端状态最小集合：

- `selectedFile`
- `preview`
- `importError`
- `importSummary`
- `conflictStrategy`
- `validating`
- `importing`

交互约束：

- 未校验前不能导入
- 有阻断性错误时不能导入
- 存在系统冲突时必须先选择策略
- `mark_conflicts` 的结果文案必须明确说明“未执行导入”

## 10. 后端结构

建议新增独立的排班导入模块，而不是复用值班人员导入模块：

- `src/lib/imports/schedule-import-template.ts`
  - 生成模板
- `src/lib/imports/schedule-import.ts`
  - 负责文件解析、预检、标准化数据输出
- `src/app/actions/schedule-import.ts` 或并入 `src/app/actions/schedule.ts`
  - 提供下载模板、预检、执行导入的 server actions

推荐保留与现有人员导入类似的 action 形态：

- `downloadScheduleTemplateAction`
- `previewScheduleImportAction`
- `importScheduleAction`

导入执行逻辑：

- 重新基于文件内容执行一次服务端预检，避免依赖前端状态
- 如果存在阻断性错误，直接返回失败和最新预检结果
- 根据策略执行：
  - `skip`：只处理无冲突记录
  - `overwrite`：处理全部可导入记录，冲突记录通过 `setSchedule()` 覆盖
  - `mark_conflicts`：直接返回统计和冲突详情，不写库

## 11. 审计日志

需要记录排班导入操作，至少包含：

- 操作类型
- 操作人
- 采用的冲突策略
- 成功导入数
- 跳过数
- 覆盖数
- 冲突数
- 失败数

如果 `mark_conflicts` 未执行落库，也应记日志，但说明为“仅生成冲突清单，未导入”。

## 12. 测试策略

### 单元/服务层

- 模板表头校验通过与失败
- 非法文件返回解析错误
- 日期格式非法
- 文件内日期重复
- 值班人员姓名无法匹配
- 系统已有排班被识别为冲突
- `skip`、`overwrite`、`mark_conflicts` 三种策略行为正确

### Playwright

- 排班页可打开“导入排班”入口
- 可下载模板
- 上传非法文件时显示错误
- 存在系统冲突时可选择策略
- `skip` 仅导入无冲突数据
- `overwrite` 会覆盖已有日期排班
- `mark_conflicts` 仅展示冲突结果且不修改排班

## 13. 风险与控制

- 重名值班人员会导致按姓名匹配歧义
  - 首版假定姓名唯一，若仓库后续支持重名，再扩展模板字段
- 导入逻辑和当前排班写入逻辑耦合
  - 通过独立的导入服务封装预检和策略分支，避免把复杂判断塞进页面组件
- 大文件导入可能带来响应时间上升
  - 首版不做异步任务化，沿用当前同步导入模式

## 14. 实施建议

建议按以下顺序实现：

1. 定义排班导入类型与模板生成器
2. 实现预检服务与三种策略
3. 增加 server actions
4. 增加排班页入口与导入弹窗
5. 补单元测试与 Playwright 回归

该顺序可以保证先把数据规则收敛，再接 UI，减少返工。
