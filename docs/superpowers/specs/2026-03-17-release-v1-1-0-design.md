# v1.1.0 发布设计

## 背景

当前仓库最新 tag 已经是 `v1.0.2`，但 `package.json` 版本仍为 `0.1.0`。系统页面的版本展示逻辑采用“优先读取 git tag，拿不到时回退 `package.json.version`”，因此代码内版本与发布版本已经发生分叉。

本次目标是发布 `v1.1.0`，并把仓库内与版本号相关的内容统一，避免后续继续出现：

- Git tag / GitHub Release 为 `v1.x.x`
- `package.json` 仍停留在旧值
- README 与当前发布状态不一致

## 目标

- 将代码版本统一到 `1.1.0`
- 创建 `v1.1.0` tag
- 创建对应 GitHub Release
- 同步更新 README 中与当前版本或发布说明直接相关的内容

## 非目标

- 不重构现有版本展示实现
- 不改动业务功能
- 不额外调整部署工作流
- 不顺带重写整份 README

## 方案选择

### 方案一：版本全面统一后再发布

先更新代码中的版本号和文档，再创建 `v1.1.0` tag 与 GitHub Release。

优点：

- `package.json`、页面展示、tag、release 一致
- 后续维护最简单
- 符合当前仓库“tag 优先、代码版本回退”的设计

缺点：

- 需要同步几处版本相关文件

### 方案二：只打 tag 和 release，不改代码版本

优点：

- 操作最快

缺点：

- 继续保留版本分叉
- 页面在无 tag 环境下仍会展示旧版本

### 方案三：只改代码版本，不打 tag

优点：

- 本地代码版本统一

缺点：

- 不构成完整发布
- GitHub Release 和 release 驱动部署链路无法闭环

## 最终设计

采用方案一，先统一仓库内版本信息，再发布 `v1.1.0`。

### 文件变更范围

#### `package.json`

- 将根包 `version` 更新为 `1.1.0`

#### `package-lock.json`

- 同步根包版本到 `1.1.0`
- 只改锁文件顶部根版本字段，不做依赖升级

#### `README.md`

- 仅更新与当前版本和发布状态直接相关的内容
- 不做与本次发布无关的文案重写

### Tag 与 Release 规则

- 新 tag 命名为 `v1.1.0`
- GitHub Release 标题使用 `v1.1.0`
- Release notes 使用中文

### Release 内容结构

Release notes 采用最小结构：

1. 本版本新增与改进
2. 本版本修复
3. 版本统一说明

建议覆盖的内容包括：

- 系统标题下方显示版本号
- 排班导入能力
- 值班人员管理页支持调整已有人员顺序
- 排序回滚保护和空格筛选判定修复
- 代码版本统一到 `1.1.0`

## 验证策略

- 确认 `package.json` 与 `package-lock.json` 根版本一致为 `1.1.0`
- 运行 `npm run lint`
- 确认本地 tag 为 `v1.1.0`
- 确认 GitHub Release 关联到 `v1.1.0`

## 风险与约束

- 如果在打 tag 前未同步 `package.json`，后续无 tag 环境仍会显示错误版本
- 如果 README 修改范围过大，会把一次简单发布变成内容编辑任务，因此本次只做最小必要更新

## 实施原则

- KISS：只做版本统一、文档同步和发布动作
- YAGNI：不顺手改发布流程、不扩展新文档结构
- DRY：统一所有版本来源，避免多个版本号长期分叉
- SOLID：保持现有版本解析模块不变，只修正其输入来源
