# Issue #24 第三期设计：值班人员管理页搜索高亮

## 背景

值班人员管理页已经支持按姓名搜索过滤，但当前结果列表只做“筛掉不匹配项”，不会在结果中明确标出命中位置。  
这会带来两个问题：

- 搜索结果较多时，用户需要自己再扫一遍名字才能确认命中原因
- 在相似姓名或部分匹配场景下，搜索反馈不够直观

## 目标

本次只完成 `#24` 的第三期能力：

1. 在值班人员管理页中，为搜索命中的姓名文本增加高亮
2. 保持现有搜索过滤逻辑不变，只增强视觉反馈

## 非目标

本次明确不包含：

- 日志页、排班页、导入页等其他搜索框的高亮
- 备注、单位、类别字段高亮
- 模糊匹配、拼音匹配、正则匹配
- 服务端搜索逻辑调整

## 行为规则

- 仅在值班人员管理页生效
- 当搜索框有非空关键词时：
  - 列表中匹配的姓名文本高亮显示
- 当搜索为空或只有空格时：
  - 恢复普通姓名显示
- 高亮只改变展示，不改变当前筛选逻辑与结果数量

## 实现设计

### 1. 数据来源

继续复用 [src/components/DutyUserManagement.tsx](/home/z/scheduling/src/components/DutyUserManagement.tsx) 中现有的 `filters.search`：

- 不增加新的本地状态
- 与当前筛选逻辑保持一致，先做 `trim()`

### 2. 组件落点

主要改造：

- [src/components/duty-users/DutyUserList.tsx](/home/z/scheduling/src/components/duty-users/DutyUserList.tsx)

新增一个极小的展示组件，例如：

- `HighlightedText`

职责：

- 接收原始文本
- 接收关键词
- 将命中片段包裹为高亮样式

这个组件只负责字符串切分和渲染，不承担过滤逻辑，保持单一职责。

### 3. 高亮范围

首版只高亮：

- 值班人员姓名

不扩展到：

- 所属单位
- 人员类别
- 备注

原因是这些字段当前不是 issue 的核心诉求，提前扩展只会增加视觉噪音。

### 4. 样式方向

高亮样式要求：

- 与当前主题兼容
- 明显但不过度抢眼
- 不破坏列表现有层级

建议使用轻量背景高亮，而不是强烈荧光色块。

## 测试策略

### Playwright

- 输入关键词后，匹配人员姓名中的命中片段出现高亮标记
- 清空搜索后，高亮消失
- 空格输入不会触发误高亮

### 回归

- 现有搜索过滤行为不变
- 现有拖拽排序、编辑、启停、删除入口不受影响

## 方案取舍

本次选择“只做值班人员管理页姓名高亮”，原因如下：

- 与 issue 原文最贴合
- 实现边界清晰，风险低
- 能作为 `#24` 的收尾项，不再继续扩散范围

这符合：

- KISS：只补最直接的视觉反馈
- DRY：复用现有搜索状态
- YAGNI：不提前把所有搜索框都统一高亮
