# Issue #22 设计：移动端月历视图仅显示单月

## 背景

当前月历视图在所有屏幕尺寸下都同时显示当前月和下个月。

这在桌面端没有问题，但在 `640px` 以下的小屏幕上会带来两个直接问题：

1. 双月并排导致单元格宽度明显变窄。
2. 移动端下姓名模式、换班摘要和交互按钮更容易显得拥挤。

`#22` 里包含多个方向：

1. 触摸目标优化
2. 手势切月
3. 小屏幕仅显示单月

这三项不应一次混做。本次只处理第三项。

## 本次范围

1. `640px` 以下月历视图只显示一个月份面板。
2. 为单月视图补明确的月份切换按钮。
3. `640px` 及以上继续保持双月视图。

明确不包含：

1. 不增加左右滑动切月手势。
2. 不改触摸热区和点击弹层详情。
3. 不改列表视图、打印页、导出页。
4. 不改排班生成、拖拽、移动、换班和日志逻辑。

## 目标

1. 小屏幕下月历宽度集中给一个月，减少挤压。
2. 移动端用户仍能顺畅浏览前后月份。
3. 桌面端现有双月使用习惯不受影响。

## 非目标

1. 不重做日历单元格样式。
2. 不引入新的页面模式或本地持久化。
3. 不做“移动端和桌面端完全两套月历组件”。

## 方案对比

### 方案 A：按断点切换为移动端单月、桌面端双月

移动端只渲染当前月；桌面端继续渲染当前月和下个月。

优点：

- 最符合 issue 原文“仅显示单月视图”。
- 只改布局层，不动数据和交互逻辑。
- 复用现有月份状态与 `MonthCalendar` 组件。

缺点：

- 需要补清晰的上月 / 下月按钮，避免移动端丢失第二个月入口。

### 方案 B：始终渲染双月，但移动端横向滚动一次看一月

优点：

- 代码表面复用更多。

缺点：

- 本质仍是双月布局，只是藏起一部分。
- 会给拖拽、焦点和滚动位置带来额外复杂度。
- 与“仅显示单月”不完全一致。

### 方案 C：继续双月，只压缩样式

优点：

- 改动最少。

缺点：

- 不能真正解决挤压问题。
- 已被近期移动端姓名拥挤反馈证明不可行。

## 选型

采用方案 A。

原因：

- 它能以最小复杂度解决最核心的可用性问题。
- 只需要在 `CalendarView` 的布局层按断点决定渲染一个还是两个 `MonthCalendar`。
- 后续如果要补手势切月，也可以在单月布局之上增量实现。

## 行为设计

### 小于 `640px`

1. 只显示当前月。
2. 顶部显示当前月标题。
3. 提供：
   - `上个月`
   - `下个月`
4. 点击后以一个月为步长切换 `currentMonth`。

### 大于等于 `640px`

1. 继续显示当前月和下个月。
2. 月份切换按钮仍然可用。
3. 每次切换推动 `currentMonth` 前后移动一个月，第二个月自动跟随。

## 组件设计

### CalendarView

这是本次唯一主要改动点。

当前它已经有：

1. `currentMonth`
2. `nextMonth`
3. `MonthCalendar`
4. 加载当前月和下个月排班数据的逻辑

改造方式：

1. 新增一个轻量的客户端断点状态，例如 `isSingleMonthMobile`
2. 用它控制月历面板渲染：
   - 移动端：只渲染 `currentMonth`
   - 桌面端：渲染 `currentMonth` + `nextMonth`
3. 保留现有数据加载范围，仍然加载当前月和下个月，避免切换到桌面或翻月后立刻再补逻辑

### MonthCalendar

不拆、不重写。

原因：

- 它已经足够单一，只负责渲染某一个月
- 单月 / 双月只是父层布局问题

## 状态与数据流

### 新状态

新增一个只读布局状态：

- `isMobileSingleMonthLayout`

来源：

- 客户端 `matchMedia('(max-width: 639px)')`

职责：

- 只决定渲染几个月份面板

### 现有状态保留

以下状态不变：

1. `currentMonth`
2. `displayMode`
3. `selectedDate`
4. `selectedDates`
5. `dragDate`
6. `moveSourceDate`

## 测试策略

### Playwright 回归

新增或扩展现有日历测试，覆盖：

1. 移动端视口下只显示一个月份标题
2. 点击 `下个月` 后，可见月份正确切换
3. 桌面端视口下仍显示连续两个月

### 回归范围

至少保证：

1. 现有移动端移动排班用例继续通过
2. 现有桌面端导航和显示模式切换用例继续通过

## 风险与控制

### 风险 1：断点状态和 SSR 初始渲染不一致

控制：

- 保持当前组件客户端渲染语义
- 在挂载后再根据 `matchMedia` 修正布局

### 风险 2：月份切换后数据范围不足

控制：

- 保留当前“加载当前月和下个月”策略
- 不在这次顺手改数据请求范围

### 风险 3：移动端只显示一个月后，用户找不到导航

控制：

- 把 `上个月 / 下个月` 按钮放在月份区域上方，使用明确中文文案或图标+文案

## 实施顺序建议

1. 先补移动端单月和桌面端双月的失败测试
2. 再改 `CalendarView` 的布局分支和月份导航
3. 最后跑移动端、桌面端联合回归
