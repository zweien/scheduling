# Issue #32 设计：支持月历式值班表模板导入

## 背景

当前系统已经支持标准行式排班导入，但其他部门提供的值班表是月历式 `.xlsx` 模板：每周两行，上行为日期，下行为值班员，首周和末周可能不是完整 7 列。现有导入逻辑无法直接解析这类模板，因此需要在保留现有导入能力的前提下新增月历式模板导入。

## 目标

1. 在现有“导入排班”面板中保留并切换两种模板：
   - 标准模板
   - 月历模板
2. 支持上传并解析月历式值班表 `.xlsx`
3. 将月历结构转换为标准排班记录，再复用现有预检、冲突处理和导入逻辑
4. 对日期解析异常、人名无法匹配、模板结构错误给出明确提示
5. 保持现有标准模板导入行为不回归

## 非目标

1. 不做模板类型自动识别
2. 不支持长按/移动端单独导入交互
3. 不支持跨月月历模板
4. 不支持从月历模板读取备注等扩展字段
5. 不新建独立的“月历导入”页面

## 方案对比

### 方案 A：在现有导入排班面板中增加模板类型切换

优点：

- 用户心智最统一
- 标准模板和月历模板共用同一套预检、冲突策略和导入结果展示
- 复用现有导入对话框和服务层，范围最可控

缺点：

- 导入逻辑需要按模板类型分支

### 方案 B：新增单独“月历式导入”入口

优点：

- 实现隔离

缺点：

- 同类功能被拆成两个入口
- 维护成本更高

### 方案 C：上传后自动识别模板类型

优点：

- 少一步用户选择

缺点：

- 模板识别脆弱
- 月历模板结构本身就不规整，误判风险高

## 结论

采用方案 A：保留现有“导入排班”面板，在面板中增加模板类型切换，月历模板解析后转换成现有标准导入行结构，再复用现有预检与导入链路。

## 用户交互设计

### 入口

入口继续保留在当前“导入排班”对话框，不新增页面。

### 模板类型切换

在导入对话框中新增模板类型选择，至少包含：

- `标准模板`
- `月历模板`

界面规则：

- 用户必须显式选择模板类型
- 下载模板按钮和说明文案随类型切换
- 上传文件后的解析逻辑严格按当前所选类型执行

### 下载模板

导入面板中根据模板类型显示对应下载入口：

- 标准模板：继续沿用当前模板
- 月历模板：新增月历式模板下载

### 上传与预检

无论哪种模板，都继续采用两阶段模式：

1. 上传文件
2. 预检并展示结果
3. 用户确认后执行导入

月历模板若存在结构错误、日期错误或值班员匹配错误，必须在预检阶段拦住，不允许直接导入。

## 月历模板结构定义

### 基本结构

月历模板的核心格式如下：

- 前面可能有空行
- 存在一个标题行，例如：`2026年3月XXX值班表`
- 存在星期头行：`周一 ~ 周日`
- 每周占 2 行：
  - 上行：日期
  - 下行：值班员

### 列结构

- 首周有效列数可能是 `1..7`
- 末周有效列数也可能是 `1..7`
- 中间周一般为完整 7 列
- 左侧辅助标签列如“日期”“值班员”可以用于弱校验，但不作为唯一识别依据

### 日期格式

模板中的日期单元格可能为：

1. Excel 序列号
2. Excel 日期值
3. 已格式化为可读日期的单元格

解析后统一转换为 `YYYY-MM-DD`。

## 解析设计

### 解析器职责

新增独立解析器文件，例如：

- `src/lib/imports/calendar-schedule-import.ts`

职责：

- 从月历 `.xlsx` 读取工作表
- 定位星期头和周块
- 将二维月历结构展开为标准排班行

解析器不负责：

- 冲突策略
- 数据库写入
- UI 展示

### 输出结构

月历模板解析后的结果应尽量复用现有标准导入行结构，例如：

- `date`
- `userName`
- `isManual`
- `notes`

规则：

- `isManual` 固定为 `true`
- `notes` 首版固定为空字符串

原因：

- 月历模板本质上代表外部人工已排好的结果
- 首版不扩展备注映射，避免过度设计

## 预检规则

### 1. 模板结构错误

需要识别并报错：

- 找不到星期头
- 周块不是成对的日期行/值班员行
- 解析出的有效日期列与值班员列错位

### 2. 日期错误

需要识别并报错：

- 单元格不是合法日期
- 同一日期重复出现
- 解析出的日期不属于同一个自然月

首版只支持单月模板，跨月视为错误。

### 3. 值班员错误

需要识别并报错：

- 姓名为空
- 姓名无法匹配到系统内值班人员
- 匹配到停用人员

### 4. 系统冲突

月历模板解析成功后，继续沿用现有排班导入的冲突策略：

- `skip`
- `overwrite`
- `mark_conflicts`

不为月历模板单独设计新的冲突处理逻辑。

## 服务层设计

### 模板类型枚举

现有导入服务增加模板类型参数，例如：

- `standard`
- `calendar`

### 编排流程

#### 标准模板

继续走现有逻辑。

#### 月历模板

1. 解析器读取工作簿并展开为标准行记录
2. 将结果传入现有预检逻辑
3. 若预检通过，继续走现有导入动作

这样可以最大化复用：

- 冲突统计
- 预检错误结构
- 导入结果统计

## 界面层设计

### 导入对话框

修改当前排班导入对话框：

- 增加模板类型切换控件
- 增加月历模板说明
- 增加月历模板下载按钮

标准模板与月历模板共用：

- 文件选择
- 预检结果区域
- 冲突策略选择
- 导入执行按钮

### 文案

月历模板模式下，应清楚提示：

- 文件必须为单月月历值班表
- 每周两行：上日期、下值班员
- 首周和末周可以不完整

## 日志与导入语义

导入成功后继续沿用现有排班导入日志行为，不新增独立日志动作。

原因：

- 模板来源不同，但业务结果仍然是“导入排班”
- 没必要为首版增加额外日志类型

如果后续确实需要区分模板来源，可再扩展日志 `target` 或 `new_value` 文案。

## 测试策略

### Node 测试

新增解析器层测试，覆盖：

1. 月历模板正常解析
2. 首周不完整可解析
3. 末周不完整可解析
4. Excel 序列号转换正确
5. 重复日期报错
6. 跨月日期报错
7. 空姓名或无法匹配姓名报错

### Playwright

覆盖以下代表性流程：

1. 导入对话框可切换模板类型
2. 月历模板模式下显示对应下载说明
3. 上传合法月历模板后能看到预检结果
4. 预检通过后可成功导入
5. 上传非法模板时显示明确错误

## 风险与控制

### 风险 1：月历模板结构不稳定

控制：

- 不做自动识别
- 要求用户显式选择“月历模板”
- 解析器按“星期头 + 周块”结构做宽松但可验证的解析

### 风险 2：新增模板类型后污染现有标准模板逻辑

控制：

- 标准模板逻辑不重写，只做显式分支
- 为标准模板保留现有回归测试

### 风险 3：日期解析依赖 Excel 单元格类型

控制：

- 统一封装日期解析函数
- 同时兼容 Excel 序列号和日期对象/格式化值

## 预期产出

1. 导入排班面板支持标准模板与月历模板切换
2. 系统可解析单月月历式值班表 `.xlsx`
3. 解析结果复用现有排班导入预检、冲突处理和导入流程
4. 月历模板导入具备清晰的错误提示和回归测试
