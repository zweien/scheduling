# 页面标题下方显示系统版本设计

日期：2026-03-17

## 1. 目标

在系统页面顶部标题区域下方显示当前系统版本号，便于用户在日常使用、问题反馈和发布确认时快速识别当前运行版本。

本次设计的目标是：

- 页面稳定展示版本号
- 版本值尽量与实际发布 tag 保持一致
- 在无法读取 tag 时提供安全回退
- 不影响现有 Header 布局和主要操作区域

## 2. 范围

### 本次包含

- 在顶部标题区域下方展示版本号
- 优先使用 `git tag` 作为版本来源
- 无法获取 tag 时回退到 `package.json.version`
- 对版本字符串做统一格式化，例如 `v1.2.3`
- 补充最小测试，确保版本来源和页面展示可验证

### 本次不包含

- 改造完整发布流程
- 新增版本历史页面
- 在页面中展示 commit hash、构建时间或环境信息
- 引入数据库或配置中心管理版本号

## 3. 设计原则

- Header 只负责展示，不负责解析版本来源
- 版本解析逻辑独立封装，避免散落在页面组件中
- 构建时注入优先于运行时执行命令，减少运行环境依赖
- 保持最小改动，不扩大到发版系统重构

## 4. 版本来源策略

版本展示采用以下优先级：

1. 优先读取构建时注入的 `APP_VERSION`
2. 如果 `APP_VERSION` 不存在，则回退到 `package.json.version`
3. 如果仍不可用，则回退为固定占位值，例如 `v0.0.0-dev`

其中：

- `APP_VERSION` 的来源为启动/构建时执行 `git describe --tags --abbrev=0`
- 如果当前仓库没有 tag，或者部署环境无法访问 `.git`，则自动回退

## 5. 实现结构

建议新增一个独立模块：

- `src/lib/app-version.ts`

职责：

- 读取 `process.env.APP_VERSION`
- 回退到 `package.json.version`
- 统一输出可展示的版本字符串

页面展示组件继续使用现有：

- `src/components/Header.tsx`

Header 中只做一件事：

- 在“值班排班系统”标题下方增加一行弱化样式的版本文本

## 6. Next.js 集成方式

在 `next.config.ts` 中注入 `APP_VERSION`：

- 启动或构建时尝试执行 `git describe --tags --abbrev=0`
- 执行成功则将结果写入环境变量
- 执行失败则不抛出构建错误，交给应用层回退

这样做的原因：

- 避免在 React 组件中运行 shell 命令
- 避免页面运行时依赖 `.git`
- 更适合部署环境和生产构建

## 7. UI 展示规则

展示位置：

- 顶部标题区域
- 主标题“值班排班系统”下方

展示形式：

- 示例：`版本 v1.2.3`
- 字号应小于主标题
- 颜色使用弱化文本色，不与主标题竞争层级

响应式要求：

- 桌面端保持可见
- 移动端保持可见
- 不能因为新增版本文本导致 Header 操作区明显挤压或换行异常

## 8. 失败与回退处理

当 `git tag` 不可获取时：

- 不阻塞本地开发
- 不阻塞构建
- 自动回退到 `package.json.version`

当 `package.json.version` 也异常时：

- 返回默认值 `v0.0.0-dev`

页面不应出现：

- 空白版本区域
- `undefined`
- 未格式化的原始值

## 9. 测试策略

### 单元测试

覆盖：

- `APP_VERSION` 存在时优先返回该值
- `APP_VERSION` 缺失时回退到 `package.json.version`
- 输出格式始终带 `v`

### 页面回归

覆盖：

- 登录后 Header 区域能看到版本文本
- 主要页面仍能正常显示标题与导航

## 10. 风险与控制

- 部署环境无法读取 `.git`
  - 通过构建期注入 + 回退机制规避
- tag 命名不规范
  - 应用层统一做 `v` 前缀格式化
- Header 高度变化影响布局
  - 仅增加一行弱化文本，必要时微调标题区 spacing

## 11. 实施建议

建议按以下顺序实现：

1. 新增版本解析工具模块
2. 在 `next.config.ts` 中注入 `APP_VERSION`
3. 在 `Header.tsx` 中展示版本文本
4. 补单元测试和页面回归

这样可以先把版本来源稳定下来，再处理 UI 展示，减少返工。
